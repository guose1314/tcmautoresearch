# 步骤 5：验证与溯源链条可视化执行总结 (2026-04-25)

## 核心任务目标
满足“同类药方在不同文献分布”的追踪诉求，完善并固化了针对 GraphRAG 环节的集成测试，并强制 LLM 输出结论时的溯源引用。系统成功实现：
1. 流转测试与断言：验证图谱中某一古方（如“四物汤”）的内容抓取及跨文献比较分析。
2. 链条追溯接口：解析大模型附带的引用 (Citations) 并映射回 Neo4j 数据库原文 ID `lit_id`。
3. 脉络可视化：以 `文献A版本 -> 药方X -> 药味组合1` 到 `文献B版本...` 的方式梳理脉络差异，支持未来直接上 Dashboard。

## 详细实施信息

### 1. 开发 GraphRAG 脉络追踪引擎 (`src/analysis/evolution_tracer.py`)
提取图数据的 Graph-RAG 构建引擎：
- 核心实体：`GraphEvolutionTracer`、`PulseDifferenceAnalysisResult`
- 从 Neo4j 图数据库查询包含此方的全部文献及具体的药材配伍 (`MATCH (p:Prescription {name: $formula})-[:APPEARS_IN]->(l:Literature)...`)。
- 执行严格的 prompt-engineering 与约束：“你必须输出 JSON 并且 `citations` 需要匹配文献标题”。
- 系统化拦截并将大模型的自然语言引用，反向链接到精确的 `neo4j_node_id`，为未来 Dashboard 树形图渲染提供锚点数据支持。

### 2. 构建专属集成测试用例 (`tests/integration_tests/test_graph_rag_traceability.py`)
模拟完整的 “前端发起查询 -> 图谱召回 -> 送入大模型知识融合 -> 溯源提取与验证” 链路：
- **流输入**：搜索 "四物汤" 在《局方》与《仙授理伤续断秘方》两部文献中配伍。
- **流验证**：拦截 GraphRAG 调用，并验证返回的 JSON 解析情况。
- **链条可视化**：
  ```
  局方 -> 四物汤 -> 熟地、当归、白芍、川芎
  仙授理伤续断秘方 -> 四物汤 -> 地黄、当归、白芍、川芎、白芷
  ```
- **追溯断言**：Citations 被稳健地绑定到 `lit-jf-1` 和 `lit-xs-2`。
